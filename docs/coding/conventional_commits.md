## Типы

| Тип        | Когда использовать (примеры)                                                               |
| ---------- | ------------------------------------------------------------------------------------------ |
| `feat`     | Новая функциональность для пользователя/публичного API                                     |
| `fix`      | Исправление бага (поведение стало корректнее)                                              |
| `docs`     | Документация: README, комментарии, ADR, примеры                                            |
| `style`    | Правки без изменения логики: форматирование, пробелы, точки с запятой                      |
| `refactor` | Перестройка кода без изменения внешнего поведения (переименование, разбиение, DI-рефактор) |
| `perf`     | Оптимизации производительности (ускорение запросов, снижение памяти)                       |
| `test`     | Добавление/правка тестов без кода продукта                                                 |
| `build`    | Сборка, зависимости, компиляторы, Dockerfile, bundler                                      |
| `ci`       | Скрипты/конфиги CI: GitHub Actions, GitLab CI                                            |
| `chore`    | Рутинные задачи, не влияющие на код продукта (обновить баннер, почистить мусор)            |
| `revert`   | Откат предыдущего коммита (`revert: feat(api): add v2 …`)                                  |

Допустимые дополнительные типы (выберите и задокументируйте у себя): `deps`, `security`, `release`, `ux`, `i18n` и т.п.

### Ломающие изменения (breaking)

Два способа пометить изменения как "потенциально опасные":
- В заголовке `feat(core)!: drop deprecated auth flow`
- В футере
```
BREAKING CHANGE: Removed field `age` for User.
```

### Semantic Version

- Есть `BREAKING CHANGE`/`!` - *major*
- Есть `feat` -> *minor*
- Есть `fix` (и чаще `perf`) -> *patch*
