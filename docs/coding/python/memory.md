## Malloc / Pymalloc

В python за аллокации отвечают разные инструменты:
- низкоуровневый `malloc`
- поверх него - Python Memory Manager
- внутри него - специализированные аллокаторы, включая `pymalloc`

Вызовы `malloc/free` достаточно дорогие, поэтому решили создать специальный
аллокатор `pymalloc` под нужды python:
- много маленьких объектов (<=512 байт)
- частые аллокации/деаллокации
- короткоживущие объекты

`pymalloc` решает эту проблему посредством **арен**: `malloc` выделяет большой
кусок памяти (~1 МБ для 64-bit), который разбивается на **пулы** (~4 КБ), которые,
в свою очередь, разбиваются на **блоки** (от 8 до 512 байт, кратно 8).
В пуле всегда блоки одного размера, а блоки - это наименьшая единица выдачи памяти.

Алгоритм выделения памяти следующий (примерно):

1. Определяем нужный размер
2. Ищем пулы с блоками этого размера
3. Если есть свободный блок - берем его
4. Если свободного блока нет, то создаем новый пул
5. Если нет арены, то выделяем новую арену

!!! note annotate "Иллюзия утечки"

    Важно помнить, что память после освобождения маленьких объектов возвращается не в систему, а обратно
    в пул. Это может создавать иллюзию утечки, хотя именно в этом и заключалась идея:
    быстрые аллокации и переиспользование выделенной памяти.

    Если же объект не пренадлежал `pymalloc` (>512 байт), то он возвращается в систему сразу же.


## Refcount (счётчик ссылок)

У каждого объекта есть счётчик ссылок и при создании он равен 1.
Когда счётчик падает до 0 - память объекта освобождается.

Проблема возникает, когда мы сталкиваемся с циклическими ссылками:

- `a` ссылается на `b`, а `b` ссылается на `a`
- `a` ссылается на саму себя

В циклических ссылках счётчик никогда не станет 0.

Тонкости:

- Слабые ссылки (`weakref`) не увеличивают `refcount`.
- `del obj` уменьшает количество ссылок на 1.
Когда количество ссылок станет 0 - вызовется `obj.__del__()`
([документация](https://docs.python.org/3/reference/datamodel.html#object.__del__))

## Garbage Collector (сборщик мусора)

За очистку циклических ссылок отвечает Garbage Collector (далее - GC). Он работает
поверх `refcount` и отслеживает контейнерные объекты. GC ищет "островки", на которые нет других
внешних ссылок и разрывает циклы.

Для оптимизации поиска циклических ссылок отслеживаются только **контейнерные объекты**: объекты,
которые потенциально могут создавать циклические ссылки. К ним относятся `list`, `dict`, `set`, muttable `tuple`,
пользовательские объекты и другие. Отслеживание можно проверить передав объект в `gc.is_tracked(obj)`.

### Модель поколений


GC использует несколько поколений для оптимизации очистки:
- 0 - молодое поколение. Проверяем чаще всего
умирают.
- 1 - среднее, переходное поколение. Убрано с версии 3.14.
- 2 - старшее поколение. Сюда попадают только те объекты, которые пережили очистку в 0 поколении.
Их проверяем реже, чем 0 поколение.

Поколения используются для того, чтобы не сканировать постоянно все объекты, а использовать
сканирование более точечно: младшее сканируем чаще, старшее - реже.

Алгоритм примерно такой:

- Когда объект создан, он попадает в **gen0**
- Находясь в **gen0** объекты чаще всего будут уничтожены механизмом `refcount`
- Когда отношение `allocations - deallocation` переваливает порог `gc.get_threshold()`, запускается GC
- Для каждого объекта в поколении берется его временный счётчик GC и присваивается ` gc_refs[obj] = refcount`
- Мы допускаем, что внутренние связи (связи внутри одного поколения) не имеют значения,
проходимся по всем ссылкам и вычитаем все внутренние ссылки
- Если объект содержит и другие ссылки - он остается жить, если нет - это мусор

### `__del__`

Если у зацикленных объектов есть `__del__`, то python не может определить безопасное удаление: порядок
вызова `__del__` неоднозначен, объект может ожить через побочные эффекты и прочее. `__del__` ломает предположение,
что GC может безопасно разорвать цикл и удалить объекты в любом порядке. GC считает, что лучше оставить объект
в памяти, чем выполнить потенциально опасный `__del__`. Ситуацию [пытаются наладить](https://peps.python.org/pep-0442/),
но проблема никуда не исчезла.

Вместо `__del__` лучше использовать менеджеры контекста (`with`) + явный метод `close()`.

```python
# Порядок удаления важен
class A:
    def __del__(self):
        self.other.do_something()  # Жив ли other? Не трогал ли его GC?


# Оживление объекта
#   1. refcount = 0
#   2. Вызывем __del__
#   3. refcount + 1, а память уже освобождается. UB?
saved = None

class A:
    def __del__(self):
        global saved
        saved = self  # Объект оживляет сам себя
```
